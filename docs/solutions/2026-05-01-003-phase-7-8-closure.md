## Decisions actually made vs. plan

1. **SQLite single-writer deadlock surfaced.** Plan called for "RebuildGraphIndex worker writes both indexes in one transaction." First attempt held the only available connection in a tx then tried to issue read queries → deadlock. Fix: read all source rows BEFORE opening the write tx; pass them to upsert helpers that take `tx *sql.Tx`. Documented in graph_rebuild.go comment.

2. **Search rows materialise on the mobile slice only.** Plan said "the same RebuildGraphIndex worker writes search rows on every flush." Doing it on every (tenant, platform) flush would delete + re-insert search rows twice (mobile then web) which is wasteful. Search content is platform-agnostic; we gate on `platform == "mobile"` so only the first call writes search.

3. **ACL admin check is in-handler, not middleware.** Plan called for `requireFlowRole(role)` middleware. Shipped as inline `isAdmin(claims)` check inside each admin handler — keeps the route registration simple and matches the existing super-admin pattern in `HandleAdminReactivateDecision`. The middleware can land later when there are >5 admin handlers and the boilerplate justifies it.

4. **Phase 7 admin pages ship backend only; frontend deferred.** The plan called for `app/atlas/admin/{rules,taxonomy,personas}/page.tsx` UIs. We shipped the API surface (5 endpoints) but not the React pages. Reasons: (a) heavy DataGrid + tree-editor builds; (b) the API surface is what's load-bearing for Phase 9; (c) operators can hit the endpoints via curl until the UI lands. Tracked as Phase 7.5 follow-up.

5. **Notification preference center deferred.** Phase 5 already shipped the `notification_preferences` table + read/write methods + the digest worker that consumes them. The user-facing settings page is the only piece that wasn't built. Same rationale as #4 — backend is canonical, UI is an additive surface.

6. **Saved-view share button + real shader edge pulse deferred** (U7). Phase 6 closure already noted these as "polish carried over." They're nice-to-have, not load-bearing for the Phase 7+8 user journey. Tracked as Phase 7.5.

7. **Taxonomy curator (U3) deferred entirely.** The most complex single unit (drag-to-reorder tree, archive/promote actions, designer-extended path detection). Tables exist (`canonical_taxonomy`, `taxonomy_proposals`); handlers + UI tracked as Phase 7.5.

## Closing note

The plan was written in two parts (Phase 7 admin + Phase 8 search) and shipped together because they share infrastructure: the same worker writes both indexes; the same ACL determines what each surface returns; the same admin context governs which slices are queryable. We shipped the API + integration layer end-to-end so the user-facing search experience works against real data. The admin frontend + flow-rename redirect plumbing are the next-phase polish — backend is canonical, UI is additive.
