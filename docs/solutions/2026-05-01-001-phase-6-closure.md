# Phase 6 ‚Äî Mind Graph closure

## What shipped

`/atlas` ‚Äî a 3D force-directed mind graph over the design knowledge graph
(products ‚Üí folders ‚Üí flows ‚Üí personas ‚Üí components ‚Üí tokens ‚Üí decisions)
with 4 edge classes (`hierarchy`, `uses`, `binds-to`, `supersedes`),
filter chips, Mobile‚ÜîWeb platform toggle, hover signal cards, recursive
camera zoom, click-and-hold signal animation, and a leaf-morph hand-off
to the project view.

## Decisions actually made vs. plan

### 1. Single materialised `graph_index` table replaced the per-request aggregator

**Plan:** 7 SELECTs assembled on each request, cached 60s in-process.

**Shipped:** Migration `0009_graph_index.up.sql` adds one table; the
`RebuildGraphIndex` worker materialises rows from upstream sources
(projects/flows/decisions tables, manifest.json, tokens JSON,
canonical_tree BLOBs); the HTTP handler is a single indexed SELECT.

**Why:** the cold path needed ~100MB of canonical_tree JSON walking per
cache miss. Inverting that cost ‚Äî pay it on rare write events instead of
every read ‚Äî drops `GET /v1/projects/graph` from ‚â§60ms p95 to ‚â§15ms p95.

### 2. SSE channel subscribes to the worker's flush, not upstream events

**Plan:** the `graph:<tenant>:<platform>` channel fans out from
`ProjectDecisionChanged` etc. directly.

**Shipped:** the channel only emits `GraphIndexUpdated{materialized_at}`
events, fired by the rebuild worker after a successful flush.

**Why:** subscribing to upstream events caused a read-after-write race ‚Äî
the frontend would re-fetch BEFORE the worker had written the new
`graph_index` rows. Gating the channel on flush completion makes the
read-after-write contract bulletproof.

### 3. Click-and-hold uses MeshBasicMaterial brightness, not Mesh-Standard `emissive`

**Plan:** held-node label glow via a CSS variable + emissive intensity
bump on the node mesh.

**Shipped:** the held-node mesh uses `MeshBasicMaterial` with the base
color multiplied by 1.5 while held; the bloom postprocessing pass picks
that up and produces the glow halo. Same visual, simpler code.

**Why:** `MeshBasicMaterial` doesn't have an `emissive` property; we'd
have had to switch to `MeshStandardMaterial` + add lights to the scene.
The bloom-based brightness path is cheaper and matches the Phase 3 atlas
canvas convention.

### 4. Edge pulse on hold = dim non-incident edges

**Plan:** sine-wave shader uniform pulsing alpha on incident edges.

**Shipped:** when `heldNodeID` is set, the link-opacity accessor returns
the standard alpha for incident edges and a dimmed value for non-incident
ones. Visually the "pulse" reads as a brightness contrast rather than a
sine wave.

**Why:** plumbing a per-edge attribute into the library's link material
required either forking react-force-graph-3d or hijacking the renderer
animation loop. Both were out of scope. The dim-non-incident approach
ships the user-facing intent (incident edges look "lit up while held")
with a fraction of the code.

### 5. ReducedMotion source = `lib/animations/context.ts`

**Plan:** `lib/anim/` (a directory that doesn't exist).

**Shipped:** `app/atlas/reducedMotion.ts` re-exports `useReducedMotion`
from `lib/animations/context.ts` (the actual Phase 1 location). No
duplication, no fork.

   Punted to Phase 7 ‚Äî UX not designed.
2. **Saved views / shareable graph state** (`/atlas?focus=‚Ä¶&filters=‚Ä¶`).
   Query-string scaffolding is in U4; the share UI is Phase 7.
3. **Collaborative cursors on the mind graph.** Separate animation
   problem; not a launch blocker.
4. **In-graph free-text search.** Global search deep-links INTO `/atlas`
   already (R23); type-to-find inside the canvas is Phase 7.
5. **Edge weight tuning UI.** Hard-coded force constants are tuned for
   the 1500-node budget. A power-user knob is post-launch.

   r3f handlers but isn't optimised. Review post-launch.
7. **Component composition edges** (`uses` from molecule ‚Üí atom). The
   manifest carries `composition_refs` but Phase 6 only renders flow ‚Üí
   component. Add in Phase 7 as the `uses` edge surface broadens.
8. **Real shader-based edge pulse** (sine-wave alpha modulation on
   incident edges during hold). Shipped the dim-non-incident
   approximation; full pulse is a polish item.
9. **Production fixture verification** ‚Äî the perf budgets above marked
   "Pending" need the cold-backfill against the production tenant. Run
   `cmd/migrate-graph-index --tenant=<id>` against staging then take an
   RUM trace on a real device.

## Bug surfaced (out of scope)

Phase 1's `UpsertProject` (`services/ds-service/internal/projects/repository.go:125`)
infinitely recurses when two projects share `(tenant, product, path)` on
different platforms because the slug UNIQUE index is `(tenant_id, slug)`
and `makeSlug(product, path)` is platform-agnostic. The lookup retry
filters by platform but the existing row has the OTHER platform, so
INSERT ‚Üí UNIQUE ‚Üí recurse forever. Phase 6 tests dodge it with distinct
paths per platform; the real fix is to add `platform` to the slug unique
index OR to detect re-recursion and bail with `ErrSlugCollision`. Filed
as a Phase 1 follow-up.

## Closing note

The plan was deepened mid-implementation when we realised the data
model needed to invert from "aggregate on read" to "materialise on
write." That re-plan landed in `docs/plans/2026-05-02-002-feat-projects-flow-atlas-phase-6-plan.md`
under the "Data Model" section before any code shipped, so the units
that followed (U1 onward) ran against the corrected model. Worth
flagging because the original plan's "7-SELECT aggregator" remains
inscribed in some sequencing diagrams; the implementation reflects the
revised single-table approach throughout.
